> 작성일: 2026-03-10
> 목적: 현재 운영 중인 SSP 시스템의 구조 파악 및 토큰화 난이도 평가

## 1. 현재 SSP 시스템 전체 구조

> 비유: SSP는 **놀이공원 자유이용권 팔찌**와 같아요. 자전거 타면(라이딩) 도장 찍히고, 중고 자전거 사고 팔면(거래) 도장 찍히고, 이 도장으로 놀이기구(상품 구매)를 탈 수 있어요.

### 1.1 3종 재화 체계

WB3에는 SSP 외에도 3가지 재화가 있어요:

| 재화 | 테이블 | T_MEMBER 컬럼 | 용도 |
|------|--------|---------------|------|
| **SSP** (스윗스웻포인트) | `T_MEMBER_SSP` | `SSP` | 라이딩/활동 보상, 상품 구매 차감 |
| **포인트** (POINT) | `T_MEMBER_POINT` | `POINT` | 일반 포인트 (PG결제 연동) |
| **캐시** (CASH / 라브머니) | `T_MEMBER_CASH` | `CASH` | 현금성 재화 |

**핵심:** 탄소저감과 직접 연결된 재화는 **SSP뿐**이에요.

### 2.2 라이딩 SSP 계산 공식 (가장 핵심)

```javascript
// API/api/routes/v1/user/mypage/riding.js:59-76
const getPoint = (data) => {
    const { IS_SURVEY, AVG_SPEED, MAX_SPEED, DISTANCE, END_DATE } = data;

    // 이미 설문 완료했으면 0
    if (IS_SURVEY === "Y") return 0;

    // 유효성 검증 (부정 방지)
    if (DISTANCE < 0.5) return 0;          // 0.5km 미만 제외
    if (AVG_SPEED < 5 || AVG_SPEED > 40) return 0;  // 비정상 속도 제외
    if (MAX_SPEED > 80) return 0;          // 최대속도 80km/h 초과 제외

    // 기본: 거리(km) × 10 SSP
    // 시간 경과에 따라 감소
    if (!END_DATE) return DISTANCE * 10;
    const e = moment(END_DATE);
    if (e.isAfter(어제)) return DISTANCE * 10;       // 어제 이후: 100%
    if (e.isAfter(일주일전)) return DISTANCE * 10 * 0.5;  // 일주일 이내: 50%
    return DISTANCE * 10 * 0.2;                         // 그 이후: 20%
};
```

**그리고 동시에 탄소저감량도 계산:**
```javascript
params.carbonReduce = distance ? distance * 0.2375 : 0;  // 109번째 줄
```

> **발견:** SSP 적립과 탄소저감량은 **같은 트랜잭션**에서 동시에 계산되지만, **계산 공식은 다르다.** SSP는 `거리×10`, 탄소저감은 `거리×0.2375`. 두 값은 1:1 대응 관계가 아니에요.

## 4. SSP 핵심 코드 구조

### 4.1 SSP 적립/차감 함수 (MemberController.ssp)

**파일:** `API/api/controllers/member.js:46-83`

```
호출: MemberController.ssp({ mapper, conn, idx, memIdx, type, amount, message, code })
  ↓
1. T_MEMBER에서 현재 SSP 잔액 조회 (getSsp)
2. 잔액 계산: IN이면 +, OUT이면 -
3. 마이너스 방지 체크
4. T_MEMBER.SSP 잔액 업데이트 (changeSsp)
5. T_MEMBER_SSP에 로그 기록 (addSspLog)
```

### 4.2 DB 구조

```
T_MEMBER (회원 마스터)
├── SSP      ← 현재 SSP 잔액 (단일 숫자)
├── POINT    ← 현재 포인트 잔액
└── CASH     ← 현재 캐시 잔액

T_MEMBER_SSP (SSP 거래 이력)
├── IDX           ← PK
├── MEMBER_IDX    ← 회원 FK
├── TYPE          ← 'IN' 또는 'OUT'
├── CURRENT       ← 거래 시점 잔액
├── AMOUNT        ← 거래 금액
├── BALANCE       ← 거래 후 잔액
├── MEMO          ← 메모 (JSON 형태도 있음)
├── CODE          ← 거래 사유 코드
├── REG_IDX       ← 등록자
└── REG_DATE      ← 등록일시
```

### 4.3 SSP 만료 로직

**파일:** `API/mapper/cron_member.xml:6-11`

```sql
-- 1년 전 적립분 중 아직 안 쓴 금액 = 만료 대상
SELECT
  (1년 전 적립 합계) - (전체 사용 합계) AS EXPIRED
FROM T_MEMBER
WHERE STATUS='Y'
HAVING EXPIRED > 0
```

만료 기간: **1년** (적립일로부터)

## 6. 블록체인/토큰 관련 기존 코드

**결론: 현재 코드에 블록체인 또는 토큰 관련 구현은 전혀 없다.**

`blockchain`, `web3`, `smart contract`, `ERC`, `solidity` 등의 키워드로 검색한 결과, 소스코드(.js, .java, .dart)에서 관련 구현이 발견되지 않았어요. SSP는 순수하게 **중앙집중형 DB 기반 포인트 시스템**이에요.

## 7. 토큰화 난이도 평가

### 7.1 현재 SSP 구조의 토큰화 장점

| 장점 | 설명 |
|------|------|
| **이력 추적 완벽** | T_MEMBER_SSP에 모든 IN/OUT 이력이 있어서 감사(audit) 가능 |
| **잔액 관리 단순** | T_MEMBER.SSP 단일 필드로 잔액 관리 |
| **사유 코드 체계** | CODE 필드로 적립 사유 구분 (RIDING, C2C 등) |
| **탄소저감 연결** | 라이딩 시 carbonReduce가 동시에 계산되어 연결 근거 있음 |

### 7.2 토큰화를 위해 필요한 변경사항

#### 난이도: 하 (기존 시스템 변경 없이 가능한 것)

| 작업 | 설명 |
|------|------|
| 토큰화 대상 SSP 스냅샷 | 특정 시점의 T_MEMBER_SSP 데이터를 추출하여 토큰 발행 근거로 사용 |
| 탄소저감-SSP 매핑 리포트 | carbonReduce와 SSP의 대응 관계를 증빙하는 리포트 생성 |

#### 난이도: 중 (API 수정 필요)

| 작업 | 설명 | 예상 기간 |
|------|------|----------|
| **SSP 동결(Lock) 기능** | 토큰으로 전환한 SSP를 사용 못하게 잠금 | 1~2주 |
| **토큰 전환 API** | SSP → 토큰 전환 신청/승인 엔드포인트 | 2~3주 |
| **이중 장부 방지** | SSP를 토큰으로 전환하면 SSP에서 차감되어야 함 | 1주 |
| **전환 이력 테이블** | `T_MEMBER_SSP_TOKEN` 같은 새 테이블 | 1주 |

#### 난이도: 상 (외부 시스템 연동 필요)

| 작업 | 설명 | 예상 기간 | 난이도 상세 |
|------|------|----------|-----------|
| **증권사 API 연동** | 토큰증권 발행을 위한 증권사 시스템 연동 | 2~3개월 | 증권사 API 스펙에 의존 |
| **KYC 강화** | 토큰증권 거래를 위한 본인인증 강화 | 1~2개월 | 금융실명제 요건 |
| **스마트컨트랙트 연동** | 블록체인에 SSP↔토큰 매핑 기록 | 1~2개월 | 체인 선택에 의존 |
| **거래소 연동** | NXT/KDX 장외거래소 API 연동 | 2~3개월 | 거래소 개장 후 가능 |
| **규제 준수 시스템** | AML/CFT, 거래 한도, 보고 의무 | 2~3개월 | 금융위 요건에 의존 |

## 8. 핵심 기술적 과제

### 과제 1: SSP와 토큰의 관계 정의

현재 SSP는 **1 SSP = 1원 가치가 아니에요.** 상품 구매 시 결제 수단으로 쓰이지만, 명확한 환율이 없어요.

```
현재: 라이딩 10km → 100 SSP 적립 + 2.375kg 탄소저감
질문: 토큰 1개 = SSP 몇 개? 또는 탄소저감 몇 kg?
```

**결정 필요:**
- 방안 A: SSP를 그대로 토큰화 (1 SSP = 1 토큰)
- 방안 B: 탄소저감량을 토큰화 (1kg CO₂ = 1 토큰), SSP는 별도 유지
- 방안 C: SSP 중 탄소저감 기반 적립분만 토큰화 대상

### 과제 2: 만료된 SSP의 처리

현재 SSP는 **1년 후 만료**돼요. 토큰증권은 만료 개념이 없어요.
→ 토큰 전환 시점에 만료 정책을 어떻게 할지 결정 필요

### 과제 3: SSP 사용과 토큰의 충돌

SSP가 상품 구매에 쓰이면 잔액이 줄어요. 토큰으로 전환한 SSP를 동시에 상품 구매에 쓸 수 있으면 이중 사용이 돼요.
→ 전환 시 SSP 차감(소각) 필수

## 9. 종합 난이도 판단

| 영역 | 난이도 | 설명 |
|------|--------|------|
| **기존 SSP 시스템 이해** | 하 | 구조가 단순하고 잘 정리되어 있음 |
| **SSP 데이터 추출/증빙** | 하 | 이력이 완벽하게 기록되어 있음 |
| **SSP 동결/전환 API 개발** | 중 | 기존 패턴 재사용 가능, 2~4주 |
| **증권사/거래소 연동** | **상** | 외부 의존, 3~6개월 |
| **규제 준수 시스템** | **상** | 금융 규제 요건, 법률 자문 필수 |
| **블록체인 연동** | 중~상 | 체인 선택과 스마트컨트랙트 개발 |

### 최종 결론

> **기술적으로 SSP를 토큰화하는 것 자체는 그리 어렵지 않아요.** 현재 SSP 시스템이 잘 만들어져 있어서 이력 추적, 잔액 관리, 탄소저감 연결이 모두 가능해요.
>
> **진짜 어려운 건 기술이 아니라 규제와 비즈니스에요:**
> 1. 금융위원회의 STO 발행 인허가
> 2. 증권사 파트너십 확보
> 3. SSP ↔ 토큰 간 가치 매핑 설계
> 4. 토큰 이코노미 (인플레이션, 소각, 수요-공급 균형)

## 10. 단계별 추천 접근법

### Step 1: 지금 바로 할 수 있는 것 (0원, 0개월)
- 현재 SSP 적립/사용 이력 데이터를 정리하여 **"이만큼의 탄소저감이 SSP로 보상되었다"** 증빙 준비
- 탄소저감량(CARBON_REDUCE)과 SSP의 대응 관계 리포트 생성

### Step 2: SSP 시스템 보강 (1~2개월)
- SSP 적립 시 탄소저감 기반인지 아닌지 구분하는 **플래그 추가** (현재는 CODE로 간접 구분)
- SSP 동결(Lock)/전환(Convert) API 개발
- 전환 이력 테이블 생성

### Step 3: 외부 연동 (3~6개월)
- 증권사 파트너 확정 후 API 연동
- 거래소(NXT/KDX) 상장 요건 확인 및 준비
- 블록체인 기록 연동 (선택)
