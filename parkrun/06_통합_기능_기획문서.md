## 0. 문서 개요

이 문서는 아래 5가지 요소를 **하나의 통합 시스템**으로 설계한 기능 기획서입니다.

| # | 요소 | 출처 | 핵심 |
|---|------|------|------|
| 1 | **파크런 이벤트** | 호주 파크런 모델 | 매주 정기 무료 이벤트 (러닝 + 자전거) |
| 2 | **구간 랭킹** | 스트라바 세그먼트 | GPS 기반 자동 순위, KOM/QOM |

## 1. 기능 모듈 구성

전체 시스템을 **6개 기능 모듈**로 나눕니다.

| 모듈 | 기능 | 우선순위 | 의존성 |
|------|------|---------|--------|
| **M1. GPX 세그먼트 관리** | GPX 파일 임포트, 세그먼트 생성/관리 | P0 (최우선) | 없음 (기반 모듈) |

| **M5. SSP 확장** | 세그먼트 완주 보상, 스탬프 연동 | P1 | M2, 기존 SSP |

## 2. M1. GPX 세그먼트 관리

### 2.1 개요

GPX(GPS Exchange Format) 파일을 시스템에 임포트하여 **세그먼트(구간)**으로 변환하고 관리하는 기반 모듈입니다. 관광공사 대표코스 60선의 GPX 파일이 1차 데이터 소스입니다.

### 2.2 GPX 파일 구조 (참고)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<gpx version="1.1">
  <trk>
    <name>여의도 한강 순환코스</name>
    <trkseg>
      <trkpt lat="37.5283" lon="126.9322">
        <ele>12.5</ele>
        <time>2024-01-01T09:00:00Z</time>
      </trkpt>
      <trkpt lat="37.5285" lon="126.9325">
        <ele>12.3</ele>
      </trkpt>
      <!-- ... 수백~수천 개의 좌표점 ... -->
    </trkseg>
  </trk>
</gpx>
```

### 2.3 세그먼트 데이터 모델

```
세그먼트(Segment)
├── segment_id          (PK, 고유 ID)
├── name                (코스명: "여의도 한강 순환")
├── description         (코스 설명)
├── activity_type       (ENUM: 'RUNNING', 'CYCLING')
├── source              (ENUM: 'KTO_60' | 'NATIONAL_TOUR' | 'USER' | 'ADMIN')
│                        KTO_60 = 관광공사 60선
│                        NATIONAL_TOUR = 국토종주 12선
│                        USER = 사용자 생성
│                        ADMIN = 관리자 생성
├── category            (ENUM: 'CITY' | 'SEA' | 'ISLAND' | 'RIVER' | 'MOUNTAIN'
│                              | 'CHALLENGE' | 'HISTORY' | 'HEALING')
│                        → 관광공사 60선 8개 카테고리와 동일
├── region              (지역: "서울", "강원", "제주" 등)
├── distance_km         (총 거리 km)
├── elevation_gain_m    (총 오르막 고도 m)
├── difficulty          (난이도 1~5)
├── gpx_data            (원본 GPX 데이터 저장 — JSON 변환)
├── polyline            (간소화된 경로 — 지도 표시용)
├── start_lat / start_lon  (시작점 좌표)
├── end_lat / end_lon      (끝점 좌표)
├── checkpoints         (중간 체크포인트 배열 — 스탬프 투어용)
├── is_loop             (순환 코스 여부)
├── is_active           (활성 상태)
├── created_at
├── updated_at
└── created_by          (생성자 — ADMIN or 회원 IDX)
```

### 2.4 GPX 임포트 프로세스

```
GPX 파일 업로드
    │
    ▼
1. XML 파싱 — trkpt(위도, 경도, 고도) 추출
    │
    ▼
2. 좌표 정제
   - 노이즈 제거 (이상치 GPS 포인트 필터링)
   - 포인트 간소화 (Douglas-Peucker 알고리즘)
     → 수천 개 포인트 → 수백 개로 축소 (지도 표시 성능)
   - 원본은 gpx_data에 보존
    │
    ▼
3. 메타데이터 계산
   - 총 거리: Haversine 공식으로 연속 좌표 간 거리 합산
   - 고도 변화: 양수 변화분만 합산 = elevation_gain
   - 시작/끝점 좌표 추출
   - 순환 코스 판정 (시작-끝 거리 < 200m 이면 순환)
    │
    ▼
4. 세그먼트 레코드 생성 → DB 저장
    │
    ▼
5. 관리자 검수 (이름, 카테고리, 난이도 확인)
```

### 2.5 관광공사 60선 초기 임포트

관광공사에서 GPX 파일을 일괄 수령 후 **배치 임포트**합니다.

| 작업 | 설명 | 예상 소요 |
|------|------|---------|
| GPX 파일 수령 | 관광공사 60선 GPX 파일 공식 수령 | - |
| 배치 파서 개발 | GPX → 세그먼트 변환 배치 스크립트 | 1주 |
| 메타데이터 매핑 | 코스명, 카테고리, 지역 매핑 테이블 작성 | 3일 |
| 일괄 임포트 | 60개 코스 + 국토종주 12선 일괄 등록 | 1일 |
| 검수 | 지도상 표시 확인, 시작/끝점 정확도 | 3일 |
| **합계** | | **약 2주** |

### 2.6 사용자 세그먼트 생성 (Phase 2)

스트라바와 유사하게 사용자가 직접 세그먼트를 만들 수 있는 기능입니다.

```
사용자의 기존 활동 기록 열기
    │
    ▼
지도에서 시작점/끝점을 탭하여 지정
    │
    ▼
시스템이 해당 구간을 세그먼트로 자동 추출
    │
    ▼
이름, 설명, 공개/비공개 설정
    │
    ▼
관리자 검수 후 공개 (스팸/부적절 세그먼트 방지)
```

## 3. M2. GPS 매칭 엔진

### 3.1 개요

사용자가 운동을 하면 그 GPS 경로를 세그먼트 DB와 대조하여 **"이 사람이 어떤 세그먼트를 통과했는지"**를 자동 감지하는 핵심 엔진입니다.

이것이 스트라바 세그먼트 랭킹의 근간이며, 전체 시스템의 기술적 핵심입니다.

### 3.2 매칭 알고리즘

```
사용자 활동 GPS 데이터 수신
    │
    ▼
1단계: 후보 세그먼트 필터링 (빠른 검색)
    ├── 사용자 경로의 바운딩 박스(bounding box) 계산
    ├── 세그먼트 시작점이 바운딩 박스 안에 있는 것만 후보로 선별
    └── → 수만 개 세그먼트를 수십 개로 축소
    │
    ▼
2단계: 시작점 진입 감지
    ├── 사용자 GPS 포인트가 세그먼트 시작점 반경 50m 이내 통과?
    └── → 통과하면 "시작 후보"로 마킹 + 시작 시각 기록
    │
    ▼
3단계: 경로 유사도 검증
    ├── 시작점 이후 GPS 포인트들이 세그먼트 경로를 따라가는지 검증
    ├── 방법: 세그먼트 경로 위의 체크포인트 N개를 지정
    │        → 사용자 경로가 각 체크포인트 반경 100m 이내를 순서대로 통과?
    └── → 80% 이상 체크포인트 통과 시 "매칭 중" 상태
    │
    ▼
4단계: 끝점 도달 감지
    ├── 사용자 GPS 포인트가 세그먼트 끝점 반경 50m 이내 통과?
    └── → 통과하면 끝 시각 기록 → "완주 확정"
    │
    ▼
5단계: 소요 시간 계산 + 검증
    ├── elapsed_time = 끝 시각 - 시작 시각
    ├── 이상치 필터링:
    │   ├── 평균속도 < 3km/h (러닝) or < 8km/h (자전거) → 제외
    │   ├── 평균속도 > 25km/h (러닝) or > 60km/h (자전거) → 제외
    │   └── 중간 정지 시간 > 30분 → 제외 (중간에 쉰 것으로 판단)
    └── → 유효한 완주 기록으로 저장
```

### 3.3 매칭 시점

| 시점 | 방식 | 장단점 |
|------|------|--------|
| **사후 매칭** (Phase 1) | 활동 종료 후 GPS 데이터 전체를 서버에 업로드 → 배치 매칭 | 구현 쉬움, 서버 부하 낮음, 실시간 아님 |
| **실시간 매칭** (Phase 2) | 활동 중 GPS 포인트를 스트리밍 → 서버에서 실시간 매칭 | 라이브 세그먼트 가능, 서버 부하 높음 |
| **스트라바 Sync 매칭** | 스트라바 API에서 가져온 활동 데이터로 사후 매칭 | 기존 스트라바 사용자 즉시 참여 |

**Phase 1에서는 사후 매칭 + 스트라바 Sync 매칭을 먼저 구현합니다.**

### 3.4 기존 스트라바 Sync 코드 활용

현재 라브 앱에는 이미 스트라바 연동이 구현되어 있습니다.

```
기존 코드: API/api/routes/v1/user/mypage/riding.js

GET /v1/user/mypage/riding/strava/sync
    │
    ├── 스트라바 API 호출 → 활동 목록 가져옴
    ├── 각 활동의 distance(m→km), speed(m/s→km/h) 변환
    ├── SSP 적립: MemberController.ssp({ code: 'RIDING-STRAVA', amount: distance×10 })
    └── 탄소저감: carbonReduce = distance × 0.2375
```

**확장 포인트:** 기존 스트라바 Sync 이후에 **세그먼트 매칭 단계를 추가**합니다.

```
기존 Strava Sync 플로우:
  스트라바 API 호출 → 활동 저장 → SSP 적립 → 끝

확장 플로우:
  스트라바 API 호출 → 활동 저장 → SSP 적립
      → [NEW] GPS 상세 경로 가져오기 (Strava Streams API)
      → [NEW] 세그먼트 매칭 엔진 실행
      → [NEW] 매칭된 세그먼트 완주 기록 저장
      → [NEW] 랭킹 업데이트
      → [NEW] 보너스 SSP 적립 (세그먼트 완주 보상)
      → [NEW] 스탬프/배지 확인 및 발급
```

### 3.5 세그먼트 완주 기록 데이터 모델

```
세그먼트 완주 기록 (SegmentEffort)
├── effort_id            (PK)
├── segment_id           (FK → 세그먼트)
├── member_idx           (FK → 회원)
├── activity_id          (FK → 활동 기록)
├── source               (ENUM: 'WB_APP' | 'STRAVA' | 'GARMIN')
├── start_time           (시작 시각)
├── end_time             (끝 시각)
├── elapsed_seconds      (소요 시간 — 초)
├── moving_seconds       (이동 시간 — 정지 시간 제외)
├── avg_speed_kmh        (평균 속도)
├── max_speed_kmh        (최고 속도)
├── avg_heart_rate       (평균 심박 — 웨어러블 연동 시)
├── is_personal_best     (개인 최고 기록 여부)
├── rank_at_time         (기록 당시 전체 순위)
├── ssp_earned           (이 완주로 적립된 SSP)
├── carbon_reduced_kg    (탄소저감량)
├── is_valid             (유효성 — 이상치 필터 통과 여부)
├── created_at
└── weather_condition    (날씨 정보 — 선택)
```

## 4. M3. 랭킹 시스템

### 4.1 랭킹 유형 종합

| 랭킹 유형 | 영감 | 기준 | 대상 | 갱신 주기 |
|----------|------|------|------|---------|
| **코스 킹/퀸 (KOR/QOR)** | 스트라바 KOM/QOM | 최단 시간 | 세그먼트별 남/녀 1위 | 실시간 |
| **단골 러너/라이더** | 스트라바 로컬 레전드 | 90일 내 최다 완주 횟수 | 세그먼트별 1위 | 일간 |
| **세그먼트 리더보드** | 스트라바 리더보드 | 소요 시간 순 | 세그먼트별 전체 | 실시간 |
| **파크런 이벤트 순위** | 파크런 결과 | 해당 회차 소요 시간 | 이벤트별 | 이벤트 종료 시 |
| **시즌 랭킹** | WB 오리지널 | 분기별 종합 포인트 | 전체 | 주간 |
| **크루/동호회 랭킹** | WB 오리지널 | 크루 누적 실적 | 크루별 | 주간 |

### 4.2 코스 킹/퀸 (KOR/QOR) 상세

스트라바의 KOM/QOM을 한국형으로 적용합니다.

```
새로운 세그먼트 완주 기록이 생김
    │
    ▼
해당 세그먼트의 기존 최고 기록 조회
    │
    ├── 남성인 경우 → 기존 KOR(코스 킹) 기록과 비교
    └── 여성인 경우 → 기존 QOR(코스 퀸) 기록과 비교
    │
    ▼
신기록이면?
    ├── YES → KOR/QOR 칭호 이전
    │         ├── 신기록자에게 왕관 아이콘 부여
    │         ├── 이전 보유자에게 알림 ("코스 킹 타이틀을 빼앗겼습니다!")
    │         ├── 신기록자에게 알림 ("축하합니다! 여의도 한강 코스 킹!")
    │         └── 보너스 SSP 지급 (신기록 보상)
    │
    └── NO → 일반 리더보드 순위 업데이트
```

**리더보드 필터링:**

| 필터 | 무료 | 프리미엄 |
|------|------|---------|
| 전체 순위 | O | O |
| 성별 필터 | O | O |
| 연령대 필터 (20/30/40/50+) | X | O |
| 크루별 필터 | X | O |
| 기간별 필터 (올해/올월/올주) | X | O |
| 친구 중 순위 | X | O |
| 체중별 필터 | X | O |

### 4.3 단골 러너/라이더 (로컬 레전드)

**속도가 아닌 빈도를 보상**하는 시스템입니다. 초보자도 왕관을 받을 수 있어 커뮤니티 참여를 높이는 핵심 기능입니다.

```
계산 방식:
  - 기준 기간: 최근 90일 (롤링 윈도우)
  - 지표: 해당 세그먼트 완주 횟수
  - 1위에게 월계관 아이콘 부여

예시:
  여의도 한강 순환 세그먼트
  ├── 김OO: 최근 90일 45회 완주 → 🏆 단골 라이더
  ├── 이OO: 최근 90일 38회 완주
  └── 박OO: 최근 90일 30회 완주

갱신: 매일 새벽 배치 작업으로 90일 윈도우 재계산
```

### 4.4 시즌 랭킹

분기별로 종합 성적을 집계하여 시즌 챔피언을 선정합니다.

```
시즌 포인트 계산 공식:

시즌 포인트 = Σ (세그먼트 완주 포인트)

세그먼트 완주 포인트 = 기본점수 + 순위 보너스 + 난이도 보너스

  기본점수: 완주 1회당 100점
  순위 보너스: 상위 10% → +50점, 상위 30% → +20점
  난이도 보너스: 난이도 × 10점 (1~5단계)

예시: 난이도 3 세그먼트를 상위 10%로 완주
  = 100 + 50 + 30 = 180 시즌 포인트
```

### 4.5 랭킹 데이터 모델

```
세그먼트 랭킹 (SegmentRanking)
├── ranking_id         (PK)
├── segment_id         (FK)
├── member_idx         (FK)
├── ranking_type       (ENUM: 'KOR' | 'QOR' | 'LOCAL_LEGEND' | 'SEASON')
├── rank               (순위)
├── best_effort_id     (FK → 최고 기록 effort)
├── best_elapsed_sec   (최고 기록 시간 — KOR/QOR용)
├── effort_count_90d   (90일 내 완주 횟수 — 로컬 레전드용)
├── season_points      (시즌 포인트 — 시즌 랭킹용)
├── season_id          (시즌 식별자: "2026-Q3")
├── updated_at
└── UNIQUE(segment_id, member_idx, ranking_type, season_id)
```

## 5. M4. 파크런 이벤트 시스템

### 5.1 개요

호주 파크런의 **"매주 같은 시간, 같은 장소"** 모델을 한국에 맞게 디지털화합니다. 러닝과 자전거 두 가지 타입을 지원합니다.

### 5.2 이벤트 타입

| 타입 | 기반 모델 | 주기 | 거리 | 대상 |
|------|---------|------|------|------|
| **WB 파크런 (러닝)** | 호주 파크런 | 매주 토요일 오전 8시 | 5km | 누구나 |
| **WB 바이크런 (자전거)** | 자전거 파크런 | 매주 토요일 오전 9시 | 15~30km | 누구나 |
| **특별 이벤트** | - | 비정기 | 다양 | 주최자 설정 |

### 5.3 이벤트 라이프사이클

```

   - GPS 기반 완주 자동 감지 (끝점 반경 50m 도달)

## 6. M5. SSP 확장 — 세그먼트 완주 보상

### 6.1 기존 SSP 시스템 현황

현재 라브 앱의 SSP 시스템을 정리합니다.

```
현재 SSP 적립 공식 (riding.js 기반):

const getPoint = (data) => {
    // 유효성 검증
    if (isSurvey === "Y") return 0;        // 설문 라이딩 제외
    if (distance < 0.5) return 0;          // 0.5km 미만 제외
    if (avgSpeed < 5 || avgSpeed > 40) return 0;   // 속도 이상치 제외
    if (maxSpeed > 80) return 0;           // 최고속도 이상치 제외

    // 시간 감쇠 (Time Decay)
    당일 라이딩:     distance × 10   (100%)
    1일~7일 전:     distance × 10 × 0.5  (50%)
    7일 이후:       distance × 10 × 0.2  (20%)
};

탄소저감 계산:
    carbonReduce = distance × 0.2375 (kg CO₂)

적립 코드:
    - RIDING-STRAVA   (스트라바 연동 라이딩)
    - RIDING-WB       (라브 앱 자체 라이딩)
    - RIDING-COURSE-SURVEY (코스 퀴즈 보너스)
```

> **참고:** 관광공사 제안서에서는 "1km당 60포인트"로 기술했으나, 실제 코드는 `distance × 10`입니다. 이 차이는 사업팀과 개발팀 간 확인이 필요합니다. 본 기획서에서는 **실제 코드 기준 `distance × 10`**을 기반으로 설계하되, 향후 단가 조정 가능하도록 설계합니다.

### 6.2 SSP 확장 — 세그먼트/이벤트 보상 추가

기존 SSP 적립 로직에 새로운 적립 경로를 추가합니다.

| 적립 코드 (신규) | 적립 조건 | 적립량 | 설명 |
|----------------|---------|--------|------|
| `SEGMENT-COMPLETE` | 세그먼트 완주 | distance × 10 | 기본 라이딩 SSP (기존과 동일) |
| `SEGMENT-FIRST` | 세그먼트 첫 완주 | +500 SSP | 첫 완주 보너스 |
| `SEGMENT-PR` | 개인 최고 기록 갱신 | +200 SSP | PR 갱신 보너스 |
| `SEGMENT-KOR` | 코스 킹/퀸 달성 | +1,000 SSP | 신기록 보상 |
| `PARKRUN-FINISH` | 파크런 이벤트 완주 | distance × 10 + 300 | 이벤트 완주 보너스 |
| `PARKRUN-VOLUNTEER` | 파크런 봉사 참여 | +500 SSP | 봉사 보상 |
| `STAMP-COLLECT` | 스탬프 수집 (60선) | 코스별 상이 | 스탬프 투어 보상 |
| `CHALLENGE-COMPLETE` | 챌린지 완료 | 챌린지별 상이 | 완주 캠페인 보상 |
| `LOCAL-LEGEND` | 단골 러너/라이더 달성 | +2,000 SSP/분기 | 로컬 레전드 보상 |

### 6.3 SSP 적립 플로우 (확장 후)

```
사용자 활동 완료
    │
    ▼
1. 기본 SSP 적립 (기존 로직)
   └── distance × 10 SSP (시간 감쇠 적용)
    │
    ▼
2. [NEW] 세그먼트 매칭 확인
   └── 매칭된 세그먼트가 있으면:
       ├── 첫 완주? → +500 SSP (SEGMENT-FIRST)
       ├── PR 갱신? → +200 SSP (SEGMENT-PR)
       └── KOR/QOR 달성? → +1,000 SSP (SEGMENT-KOR)
    │
    ▼
3. [NEW] 파크런 이벤트 참가 확인
   └── 이벤트 참가자이면:
       └── +300 SSP (PARKRUN-FINISH)
    │
    ▼
4. [NEW] 스탬프 투어 확인
   └── 관광공사 60선 세그먼트이면:
       ├── 스탬프 발급
       ├── 스탬프 보상 SSP 지급
       └── 마일스톤 체크 (10/30/60개)
    │
    ▼
5. 탄소저감 계산 (기존 로직)
   └── carbonReduce = distance × 0.2375
    │
    ▼
6. [NEW] ESG 대시보드 업데이트
   └── 전체 탄소저감 누적량 갱신
```

### 6.4 SSP 단가 설정 테이블 (관리자 설정 가능)

향후 단가 변경에 유연하게 대응하기 위해 **설정 테이블**로 관리합니다.

```
SSP 설정 (SSPConfig)
├── config_id
├── config_key           ("SSP_PER_KM", "SEGMENT_FIRST_BONUS", ...)
├── config_value         (10, 500, ...)
├── description          ("1km당 기본 SSP")
├── is_active
├── effective_from       (적용 시작일)
├── effective_to         (적용 종료일 — null이면 무기한)
└── updated_at

초기 설정값:
  SSP_PER_KM = 10
  SEGMENT_FIRST_BONUS = 500
  SEGMENT_PR_BONUS = 200
  SEGMENT_KOR_BONUS = 1000
  PARKRUN_FINISH_BONUS = 300
  VOLUNTEER_BONUS = 500
  LOCAL_LEGEND_BONUS = 2000
  CARBON_PER_KM = 0.2375
```

## 7. M6. 관광공사 연동 — 스탬프 투어 & ESG

### 7.1 디지털 스탬프 투어

관광공사 대표코스 60선을 기반으로 한 **디지털 수집형 콘텐츠**입니다.

### 7.1.1 스탬프 수집 메커니즘

```
관광공사 60선 세그먼트 완주
    │
    ▼
자동 스탬프 발급
├── 코스별 고유 디자인 스탬프 이미지 저장
├── 완주 일시, 소요 시간, 날씨 정보 기록
└── 앱 내 "나의 스탬프북" 에 표시
    │
    ▼
마일스톤 달성 확인
├── 10개 완주 → 브론즈 라이더 배지 + SSP 1,500P
├── 30개 완주 → 실버 라이더 배지 + SSP 5,000P
├── 60개 완주 → 골드 라이더 배지 + SSP 15,000P
└── 카테고리 컴플리트 → 카테고리 마스터 배지
    (예: 바다 코스 전체 완주 → "바다의 왕" 배지)
```

### 7.1.2 스탬프 데이터 모델

```
스탬프 (Stamp)
├── stamp_id           (PK)
├── member_idx         (FK)
├── segment_id         (FK → 관광공사 60선 세그먼트)
├── effort_id          (FK → 해당 완주 기록)
├── stamp_image_url    (스탬프 이미지 URL)
├── collected_at       (수집 일시)
├── elapsed_seconds    (완주 소요 시간)
└── weather            (날씨 정보)

배지 (Badge)
├── badge_id           (PK)
├── name               ("브론즈 라이더", "바다의 왕")
├── description
├── badge_image_url
├── badge_type         (ENUM: 'MILESTONE' | 'CATEGORY' | 'CHALLENGE' | 'SPECIAL')
├── requirement_type   (ENUM: 'STAMP_COUNT' | 'CATEGORY_COMPLETE' | 'ROUTE_COMPLETE')
├── requirement_value  (10, 30, 60 또는 카테고리 코드)

## 11. 기술적 고려사항

### 11.1 GPS 매칭 성능

세그먼트 매칭은 **모든 활동에 대해** 실행되므로 성능이 중요합니다.

| 고려사항 | 해결 방안 |
|---------|---------|
| 세그먼트 수 증가 시 매칭 시간 증가 | 바운딩 박스 사전 필터로 후보 축소 |
| GPS 정확도 편차 (도시 vs 산간) | 매칭 반경을 환경별 조정 (도시 30m, 산간 80m) |
| 실시간 매칭 서버 부하 | Phase 1은 사후 매칭만, 실시간은 Phase 3에서 WebSocket 도입 |
| 대량 동시 활동 (파크런 이벤트) | 이벤트 참가자 매칭은 이벤트 세그먼트만 대조 (후보 1개) |

### 11.2 공간 데이터 처리

| 항목 | 권장 방식 |
|------|---------|
| 좌표 저장 | MySQL POINT 타입 or 위도/경도 DECIMAL(10,7) |
| 거리 계산 | Haversine 공식 (서버 사이드) |
| 근접 검색 | 공간 인덱스 (MySQL Spatial Index or R-Tree) |
| 경로 간소화 | Douglas-Peucker 알고리즘 (epsilon=0.0001) |

> **참고:** 기존 라브 시스템이 MySQL 기반이면 MySQL Spatial Functions 활용, PostgreSQL이면 PostGIS 활용을 권장합니다.

### 11.3 스트라바 API 제약

| 항목 | 제한 |
|------|------|
| Rate Limit | 15분당 100회, 일 1,000회 (기본) |
| GPS Streams | 활동별 개별 API 호출 필요 (대량 처리 시 주의) |
| Webhook | 새 활동 등록 시 자동 알림 가능 (Push 방식 권장) |

**권장:** 스트라바 Webhook을 설정하여 사용자가 새 활동을 올리면 자동으로 세그먼트 매칭이 실행되도록 합니다. 현재 폴링 방식(`/strava/sync`)보다 효율적입니다.

### 11.4 데이터 정합성

| 시나리오 | 해결 방안 |
|---------|---------|
| 같은 활동이 스트라바+라브 앱 양쪽에서 중복 매칭 | activity source 기준 중복 체크 |
| KOR/QOR 동시 갱신 (2명이 동시에 신기록) | DB 트랜잭션 + 낙관적 잠금 |
| SSP 이중 적립 | 기존 `T_MEMBER_SSP` 트랜잭션 로그로 중복 방지 |

### B. 용어 정리

| 용어 | 설명 |
|------|------|
| 세그먼트 (Segment) | GPS 기반 특정 구간. 시작점~끝점으로 정의 |
| KOR/QOR (King/Queen of Route) | 세그먼트 남/녀 최고 기록 보유자 |
| 로컬 레전드 (Local Legend) | 90일 내 특정 세그먼트 최다 완주자 |
| 매칭 (Matching) | 사용자 GPS 경로가 세그먼트를 통과했는지 자동 판별 |
| Effort | 세그먼트 1회 통과 기록 |
| PR (Personal Record) | 개인 최고 기록 |
| SSP | Sweet Sweat Point — 라이딩 보상 포인트 |
| GPX | GPS Exchange Format — GPS 경로 데이터 표준 형식 |
| 파크런 (parkrun) | 매주 정기적으로 열리는 무료 커뮤니티 러닝/라이딩 이벤트 |
| 바운딩 박스 | GPS 좌표의 최소/최대 위경도로 만든 사각 영역 |
