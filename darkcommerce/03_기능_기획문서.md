# 깜깜이방 기능 기획문서

> 작성일: 2026-03-19
> 전제: DB는 기존 유지, 앱/웹/API는 전부 리뉴얼
> 대상: 개발자가 이 문서만 보고 구현할 수 있는 수준

---

## 1. 기능 한줄 요약

대리점이 상품을 등록하면 **라이트브라더스 이름으로만** 할인 판매되는 익명 마켓플레이스.

---

## 2. 사용자 역할 (Role)

| 역할 | 설명 | 접근 영역 |
|------|------|----------|
| **소비자** | 앱/웹에서 깜깜이방 상품을 구매하는 일반 회원 | 깜깜이방 탭, 상품 상세, 주문 |
| **딜러** | 할인 상품을 등록하는 대리점 사장님 | 딜러 전용 페이지 (상품등록, 정산조회) |
| **관리자** | 라브 운영팀. 상품 검수, 정산 처리 | 관리자 깜깜이방 메뉴 |

**기능 요구사항:**
- 현재 진행 중인 Flash Sale 상품을 **마감 임박 순**으로 정렬
- 각 상품 카드에 **실시간 카운트다운 타이머** 표시
- 오픈 예정 상품은 **이미지 블러 처리** + 오픈 시간만 표시
- 매진된 상품은 "SOLD OUT" 오버레이 + 리스트 하단으로 이동
- Pull-to-refresh로 새로고침
- 판매자 정보는 **어디에도 표시하지 않음** (라이트브라더스만 표시)

**데이터 요구사항:**
```

**입력 필드 상세:**

| 필드 | 타입 | 필수 | 유효성 검사 |
|------|------|------|-----------|
| 상품 사진 | 이미지 배열 | ✅ | 최소 3장, 최대 10장, 각 10MB 이하, JPG/PNG/WEBP |
| 카테고리 | 셀렉트 | ✅ | 완성차/부품/러닝화/의류/액세서리 |
| 브랜드 | 검색+셀렉트 | ✅ | 기존 T_BRAND에서 검색 |
| 상품명 | 텍스트 | ✅ | 5~100자 |
| 상품 상태 | 라디오 | ✅ | NEW/DISPLAY/REFURB |
| 정가 | 숫자 | ✅ | 1,000원 이상 |
| 납품가 | 숫자 | ✅ | 정가의 40% 이상 (너무 낮으면 의심) |
| 수량 | 숫자 | ✅ | 1~10 (초기에는 1개 권장) |
| 판매 기한 | 라디오 | ✅ | 48h/72h/7d |
| 상품 설명 | 텍스트 | 선택 | 최대 2,000자 |
| 배송 방법 | 라디오 | ✅ | DIRECT_SHIP/VIA_WB |

**"예상 수익 계산" 실시간 표시:**
```
소비자 판매 예상가 = 납품가 ÷ (1 - 수수료율)
소비자 할인율 = (정가 - 판매 예상가) ÷ 정가 × 100
딜러 수익 = 납품가

※ 최종 판매가는 라브가 검수 시 조정할 수 있음
```

### 6-2. 기존 테이블 연결 방식

```
[기존 T_ORDER 시스템과의 연결]

소비자가 깜깜이방 상품을 주문하면:
1. T_ORDER에 일반 주문처럼 생성 (기존 결제 플로우 그대로 사용)
2. T_ORDER_ITEM에 주문 항목 생성
3. T_DARK_ORDER에 깜깜이방 전용 정보 추가 생성

→ 결제/환불은 기존 시스템, 딜러 정산은 깜깜이방 시스템에서 처리

[기존 T_BRAND와의 연결]

딜러가 상품 등록 시 T_BRAND에서 브랜드를 검색/선택
→ T_DARK_GOODS.BRAND_IDX에 저장
→ 브랜드 관리는 기존 관리자 화면에서 그대로 사용

[기존 T_MEMBER와의 연결]

딜러도 회원가입 필요 → T_MEMBER에 계정 생성
→ T_DARK_DEALER.MEMBER_IDX로 연결
→ 로그인은 기존 인증 시스템 사용
```

## 8. 비즈니스 규칙

### 8-1. 가격 규칙

```
규칙 1: 납품가 ≥ 정가 × 40%
  → 너무 싸면 위조품 의심 → 등록 차단

규칙 2: 판매가 = 납품가 ÷ (1 - 수수료율)
  → 예: 납품가 52만, 수수료 10% → 판매가 = 52만 ÷ 0.9 ≈ 57.8만
  → 관리자가 최종 조정 가능 (단, 납품가보다 높아야 함)

규칙 3: 할인율 = (정가 - 판매가) ÷ 정가 × 100
  → 소비자에게 표시되는 할인율

규칙 4: 수수료 금액 = 판매가 - 납품가
  → 라브의 실제 수익
```

### 8-2. 상태 전이 규칙

```
[상품 상태]

PENDING ──(관리자 승인)──→ APPROVED ──(시작시간 도달)──→ ACTIVE
   │                                                      │
   │──(관리자 반려)──→ REJECTED                            │
   │                                                      │──(수량 0)──→ SOLD_OUT
   │──(딜러 취소)──→ CANCELLED                             │
                                                          │──(기한 만료)──→ EXPIRED

[주문-배송 상태]

PENDING ──(딜러 발송)──→ SHIPPED ──(배송완료)──→ DELIVERED
   │                       │                       │
   │──(기한 초과)──→ AUTO_CANCEL                   │──(구매확정)──→ CONFIRMED
                                                   │
                                                   │──(반품요청)──→ RETURN_REQ ──→ RETURNED

[정산 상태]

PENDING ──(구매확정)──→ READY ──(관리자 입금)──→ PAID
```

### 8-3. 자동화 규칙

```
규칙 A: Flash Sale 자동 시작/종료
  - 매 분 크론잡 실행
  - APPROVED 상품 중 SALE_START_AT ≤ 현재시간 → ACTIVE로 변경
  - ACTIVE 상품 중 SALE_END_AT ≤ 현재시간 → EXPIRED로 변경

규칙 B: 미발송 자동 취소
  - 매 시간 크론잡 실행
  - PENDING 상태 + SHIP_DEADLINE 초과 → 주문 자동 취소
  - 딜러에게 패널티 알림

규칙 C: 자동 구매확정
  - 배송완료 후 7일 경과 + 반품요청 없음 → 자동 구매확정
  - 정산 상태를 READY로 변경

규칙 D: 딜러 등급 자동 산정
  - 매월 1일 크론잡
  - 조건 충족 시 자동 승급 (Bronze→Silver→Gold)
```

### 8-4. 보안/익명성 규칙

```
규칙 X-1: API 응답에 딜러 정보 절대 미포함
  - 소비자 대상 모든 API에서 dealerIdx, dealerName 등 제거
  - 서버 측에서 직렬화 시 필터링

규칙 X-2: 딜러 DB 접근 권한 제한
  - T_DARK_DEALER 테이블은 admin 권한만 접근 가능
  - 딜러 본인은 자기 정보만 조회 가능

규칙 X-3: 관리자 화면에서도 딜러 코드만 표시
  - 딜러 상세를 보려면 별도 권한 필요
  - 접근 로그 기록

규칙 X-4: 배송지 마스킹
  - 딜러에게 전달하는 수령인 이름: 첫 글자만 + **
  - 연락처: 중간 4자리 ****
  - 주소: 전체 표시 (배송 필수)

규칙 X-5: 반품 주소
  - 반품 시 라브 사무실/물류센터 주소 안내
  - 딜러 주소 절대 노출하지 않음
```

```

  ├── 깜깜이방 메인 페이지 (상품 리스트 + 타이머)
  ├── 상품 상세 페이지

[Sprint 3 - 2주] 배송 + 정산
  ├── 딜러 출고 요청 화면 + 송장 입력
  ├── 배송 추적 연동
  ├── 정산 시스템 (자동 집계 + 관리자 입금 처리)
  └── 알림 시스템 연동

[Sprint 4 - 1주] QA + 런칭
  ├── 전체 플로우 테스트
  ├── 익명성 검증 (API 응답에 딜러 정보 누출 없는지)

  └── 초기 딜러 3~5곳 온보딩
