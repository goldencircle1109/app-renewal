## 1. 핵심 결론

**가능하지만 조건부입니다.** 센서만으로는 한계가 있고, GPS + GIS(지도 데이터)를 결합해야 실용적입니다.

| 질문 | 답변 |
|------|------|
| 8개 이상 교통수단 구분 가능한가? | **조건부 가능** (센서 조합 + GIS 필수) |
| 연구실 최고 정확도 | 97.7% (TCN 모델, 2022) |
| 실서비스 현실적 정확도 | 85~90% |
| 버스 vs 승용차 구분 | **가장 어려움** — GPS+GIS 없이는 60~65% |
| 지하철 vs 기차 구분 | **두 번째로 어려움** — 기압계+GIS 필요 |
| 최소 필요 센서 | 가속도계 + GPS (2개로 기본 가능) |
| 최적 센서 조합 | 가속도계 + 자이로스코프 + GPS + 기압계 (4개) |
| 배터리 소모 문제 | GPS 상시 사용 시 심각 → 듀티 사이클링 필수 |
| 실시간 vs 사후 분류 | 사후 분류가 5~10% 더 정확 |

## 2. 감지 대상 교통수단 8종과 특성

### 2-1. 교통수단별 물리적 신호 특성

| 교통수단 | GPS 속도 | 가속도 진폭 | 진동 주파수 | 정차 패턴 | 구분 난이도 |
|---------|---------|-----------|-----------|----------|-----------|
| 걷기 | 3~7 km/h | 1~2g | 1.5~2.5 Hz (보행 리듬) | 신호 대기 | ⭐ 쉬움 |
| 뛰기 | 8~15 km/h | 2~4g | 2.5~3.5 Hz (착지 충격) | 드물게 정지 | ⭐ 쉬움 |
| 자전거 | 10~35 km/h | 낮고 불규칙 | 0.5~1.5 Hz (페달링) | 신호 대기 | ⭐⭐ 보통 |
| 오토바이 | 20~80 km/h | 중간, 엔진 진동 | 20~100 Hz (엔진) | 짧은 정차 | ⭐⭐ 보통 |
| 승용차 | 20~120 km/h | 낮음, 부드러움 | 5~50 Hz (노면 진동) | 신호 대기 | ⭐⭐⭐ 어려움 |
| 버스 | 0~60 km/h | 중간, 불규칙 | 5~50 Hz + 충격 | **빈번한 정차** | ⭐⭐⭐⭐ 매우 어려움 |
| 택시 | 20~80 km/h | 승용차와 유사 | 승용차와 동일 | 승용차와 유사 | ⭐⭐⭐⭐⭐ 극도로 어려움 |
| 지하철 | 30~80 km/h | 매우 낮음 | 레일 진동 | **역 정차 규칙적** | ⭐⭐⭐ 어려움 |

> **쉽게 말하면**: 걷기/뛰기는 폰이 흔들리는 패턴이 명확해서 쉽고, 버스/승용차/택시는 "같은 도로 위 차량"이라 구분이 어렵습니다.

### 2-2. 왜 택시는 거의 구분 불가능한가?

택시와 승용차는 **같은 종류의 차량**이 **같은 도로**를 달리므로 센서 신호가 거의 동일합니다.
구분하려면 센서가 아닌 **외부 데이터** 필요:
- 카카오택시/타다 앱 연동 (탑승 여부 직접 확인)
- 결제 데이터 연동
- 사용자에게 직접 질문

→ **결론: 택시는 "승용차"와 합쳐서 하나로 분류하는 것이 현실적**

## 3. 센서별 역할과 기여도

### 3-1. 가속도계 (Accelerometer) — 핵심 센서, 기여도 60~70%

모든 스마트폰에 내장되어 있고, 교통수단 감지의 기본이 되는 센서입니다.

| 장점 | 단점 |
|------|------|
| 배터리 소모 매우 낮음 (~1% 추가) | 단독으로는 버스/승용차 구분 불가 |
| 모든 스마트폰에 내장 | 폰 위치(손/주머니/가방)에 따라 신호 변화 |
| 걷기/뛰기 구분에 탁월 | 정차 중인 차량은 판단 불가 |

**작동 원리 (쉽게)**:
- 걸을 때 → 폰이 "쿵, 쿵" 규칙적으로 흔들림 (1~2초 간격)
- 뛸 때 → 폰이 "쾅, 쾅" 더 세게, 더 자주 흔들림
- 차 안 → 폰이 거의 안 흔들리고, 도로 진동만 살짝
- 자전거 → 폰이 불규칙하게 흔들림 (페달링 리듬)

### 3-2. GPS — 속도/경로 정보, 기여도 20~30%

| 장점 | 단점 |
|------|------|
| 속도로 교통수단 대략 구분 가능 | **배터리 소모 심각** (시간당 5~10%) |
| 버스 노선 매칭에 핵심 | 지하/터널에서 **신호 없음** |
| 경로 정보로 GIS 매핑 가능 | 도심 빌딩 사이 오차 최대 20m |

**속도만으로 구분 가능한 것:**
- 0~3 km/h = 정지
- 3~7 km/h = 걷기
- 8~15 km/h = 뛰기 또는 자전거(저속)
- 15~35 km/h = 자전거 또는 도심 차량
- 35~80 km/h = 차량(종류 불명)
- 80~300 km/h = KTX/SRT 또는 고속도로

→ **속도만으로는 "차량 종류"를 구분하기 어렵습니다.** 버스도 승용차도 같은 속도대를 공유합니다.

### 3-3. 기압계 (Barometer) — 지하철 감지 핵심

| 장점 | 단점 |
|------|------|
| 지하 진입 감지 가능 (기압 변화) | 중저가 폰에는 미탑재 |
| 배터리 소모 매우 낮음 | 날씨에 의한 기압 변화와 혼동 가능 |
| 엘리베이터/계단 감지도 가능 | |

**작동 원리 (쉽게)**:
지하철역 계단을 내려가면 기압이 올라갑니다. 이 변화를 감지해서 "지하로 내려갔다" → "지하철 탈 가능성 높다"고 판단합니다.

### 3-4. 자이로스코프 — 보조 센서, 기여도 2~5%

- 폰의 회전을 감지
- 오토바이 코너링 시 독특한 회전 패턴 → 오토바이 구분에 도움
- 가속도계와 합치면 정확도 소폭 향상

### 3-5. Wi-Fi — 대중교통 직접 감지

한국에서 특히 유용한 방법입니다:

| Wi-Fi SSID | 의미 |
|-----------|------|
| `BUS_FREE_WIFI`, `버스와이파이` 류 | 시내버스 탑승 중 |
| `SKT_WiFi_PUBLIC`, `U+Zone`, `KT_GiGA_WiFi` | 지하철역/차량 내 |
| `KTX_WiFi`, `SRT_WiFi` | 고속철도 탑승 중 |

→ 버스/지하철 공공 Wi-Fi SSID를 직접 감지하면 **즉시 교통수단 확인 가능!**
(단, 모든 버스/지하철에 Wi-Fi가 있지는 않음)

### 3-6. 셀타워 (Cell Tower) 신호

- 배터리 소모 거의 없음
- 기지국 전환 속도로 이동 속도 대략 추정
- 지하철 터널에서 신호 소멸 → 지하 진입 감지
- 정확도는 낮지만, GPS 없을 때 대체 수단으로 유용

### 3-7. 센서 조합별 정확도 비교

| 센서 조합 | 정확도 | 배터리 영향 |
|---------|--------|-----------|
| 가속도계만 | 70~80% | 매우 낮음 |
| 가속도계 + GPS | 85~91% | 중간 |
| 가속도계 + GPS + 기압계 | 88~93% | 중간 |
| 가속도계 + 자이로 + GPS + 기압계 | 90~95% | 중간~높음 |
| 위 4개 + Wi-Fi + GIS 데이터 | 92~96% | 높음 |

## 4. 기존 플랫폼 API (Google / Apple)

### 4-1. Google Activity Recognition API (Android)

```
지원하는 활동:
✅ STILL (정지)
✅ WALKING (걷기)
✅ RUNNING (뛰기)
✅ ON_BICYCLE (자전거)
✅ IN_VEHICLE (차량 — 버스/승용차/택시 미구분!)
❌ 버스/지하철/기차/오토바이 개별 구분 불가
```

**한계:**
- `IN_VEHICLE` 하나로 모든 차량을 묶어버림
- 무료지만 정확도 보장 없음
- Google Play Services 필수 (중국 등 불가)

### 4-2. Apple Core Motion (iOS)

```
지원하는 활동:
✅ stationary (정지)
✅ walking (걷기)
✅ running (뛰기)
✅ cycling (자전거)
✅ automotive (차량 — 버스/지하철 미구분!)
❌ 버스/지하철/기차/오토바이 개별 구분 불가
```

**장점:** M-시리즈 코프로세서가 전담 처리 → 배터리 거의 소모 안 함, 앱 종료 후에도 수집
**한계:** 역시 모든 차량을 `automotive` 하나로 묶음

### 4-3. 플랫폼 API의 공통 한계

| 가능한 것 | 불가능한 것 |
|---------|-----------|
| 걷기 / 뛰기 / 자전거 / 차량 구분 | 버스 vs 승용차 |
| 정지 상태 감지 | 지하철 vs 기차 |
| 저전력 백그라운드 동작 | 오토바이 구분 |
| | 택시 구분 |

→ **결론: 플랫폼 API는 "차량 안에 있다"까지만 알려줌. 어떤 차량인지는 직접 구현해야 함.**

## 5. 머신러닝 접근법과 정확도

### 5-1. 알고리즘별 비교

| 알고리즘 | 정확도 | 스마트폰 실행 가능? | 난이도 |
|---------|--------|----------------|-------|
| Random Forest | 80~88% | ✅ 가능 | 쉬움 |
| SVM | 75~85% | ✅ 가능 | 쉬움 |
| CNN | 88~94% | ✅ TFLite 가능 | 보통 |
| LSTM | 90~95% | ✅ 경량화 시 | 보통 |
| CNN-BiLSTM | 93~96% | ✅ 최적화 시 | 어려움 |
| TCN | **최대 97.7%** | ✅ 병렬 처리 빠름 | 어려움 |

> **쉽게 설명**: CNN은 "사진 인식"에 쓰이는 기술인데, 센서 데이터를 "그래프 이미지"처럼 보고 패턴을 찾습니다. LSTM은 "시간 흐름"을 이해하는 기술이라, "5분간 계속 흔들렸다 → 걷기"처럼 판단합니다.

### 5-2. SHL 챌린지 (가장 권위있는 교통수단 감지 대회)

Sussex-Huawei Locomotion 챌린지는 매년 열리는 교통수단 감지 벤치마크입니다.

- **데이터셋 규모**: 2,812시간, 16,732 km
- **분류 대상**: 정지/걷기/뛰기/버스/승용차/기차/지하철
- **2023년 최고 성적**: F1 스코어 96.0%
- **실서비스 환경**: 연구실 대비 5~10% 낮아짐 (85~90%)

### 5-3. 연구실 vs 실서비스 성능 격차

| 구분 | 연구실 | 실서비스 | 격차 원인 |
|------|-------|---------|---------|
| 전체 평균 | 93~97% | 85~90% | 다양한 폰/위치 |
| 버스 vs 승용차 | 75~85% | 60~75% | 교통 정체, 도로 상태 |
| 지하철 vs 기차 | 70~80% | 65~75% | 레일 진동 유사 |
| 걷기 vs 뛰기 | 95~99% | 90~95% | 폰 위치 변화 |

→ 구글 같은 대기업도 교통수단 100% 정확 분류는 어렵습니다. Google Maps 타임라인도 승용차 감지율 57% 수준.

## 6. 가장 어려운 문제들

### 6-1. 버스 vs 승용차 (가장 어려운 과제)

**왜 어려운가?**
- 둘 다 같은 도로 위를 달림 → 비슷한 속도
- 교통 정체 시 신호가 거의 동일
- 폰 위치(시트, 손잡이, 가방)에 따라 진동 변화

**해결 방법 (효과 순):**
1. **GPS + 버스 노선 GIS 매핑** — 버스 정류장에서 규칙적 정차 패턴 확인 (가장 효과적)
2. **Wi-Fi SSID 감지** — 버스 공공와이파이 직접 감지 (한국에서 유용)
3. **정차 빈도 분석** — 버스는 승용차보다 자주 정차 (2~3분 간격)
4. **가속도 분산** — 버스가 승용차보다 진동이 큼

**현실적 정확도:**
- 가속도만 → 50~65%
- GPS 추가 → 65~75%
- GPS + GIS 버스 노선 → 75~85%

### 6-2. 지하철 vs 기차

**왜 어려운가?**
- 둘 다 레일 위를 달림 → 가속도 신호 매우 유사
- 지하에서는 GPS 신호 없음

**해결 방법:**
1. **GPS 속도** — KTX/SRT는 250~300 km/h, 지하철은 최대 80 km/h → 고속철도는 쉽게 구분
2. **기압계** — 지하 진입 시 기압 변화
3. **정차 간격** — 지하철 2~5분, 기차 10~30분 이상
4. **GIS 노선 매핑** — 지하철 노선 vs 기차 노선 위치 대조

### 6-3. 배터리 소모 문제

| 센서 | 배터리 소모 | 절감 방법 |
|------|-----------|---------|
| 가속도계 (10 Hz) | ~1%/시간 | 낮은 샘플링 속도 |
| GPS (상시) | 5~10%/시간 | **듀티 사이클링 필수** |
| GPS (30초 간격) | 3~5%/시간 | 정확도 소폭 감소 허용 |
| Wi-Fi 스캐닝 | ~3%/시간 | 주기적 스캔만 |

**배터리 효율적 아키텍처 (추천):**

```
[항상 켜짐] 가속도계 (10 Hz, 거의 무소모)
    │
    ├── "움직이지 않음" → GPS 끔 (배터리 절약)
    │
    └── "이동 감지!" → GPS 활성화 (30초 간격)
            │
            ├── 분류 완료 → GPS 끔
            └── 차량 탑승 확인 → GIS 매핑 시작 → 하차 시 종료
```

이렇게 하면 GPS를 이동 시에만 켜서 **배터리 소모를 1/3로 줄일 수 있습니다.**

### 6-4. 폰 위치 문제

같은 버스를 타도 폰을 손에 들 때, 주머니, 가방에 넣을 때 센서 신호가 완전히 다릅니다.

**해결 방법:**
- "위치 독립적" 특징 추출 (진동의 크기만 보고, 방향은 무시)
- 최신 딥러닝 모델(Feature Pyramid biLSTM)은 4개 위치에서도 95% 정확도 달성
- 주파수 분석(FFT) 기반 특징이 위치 변화에 가장 강건

## 7. 한국 특화 고려사항

### 7-1. 한국 교통 인프라 활용

한국은 대중교통 인프라가 잘 되어있어, 활용할 수 있는 데이터가 많습니다.

| 데이터 | 활용 방법 | 출처 |
|-------|---------|------|
| 버스 정류장 GPS 좌표 | 정차 위치가 정류장과 일치하면 → 버스 | 공공데이터포털 (data.go.kr) |
| 버스 노선 경로 | GPS 경로가 버스 노선과 겹치면 → 버스 | 공공데이터포털 |
| 지하철역 좌표 | 정차 위치가 지하철역이면 → 지하철 | 서울 열린데이터광장 |
| 기차역 좌표 | 정차 위치가 기차역이면 → 기차 | 코레일 공공 API |
| 카카오맵/네이버맵 API | 실시간 대중교통 노선 정보 | 카카오/네이버 |

### 7-2. 한국 공공 Wi-Fi 활용

| Wi-Fi SSID 패턴 | 교통수단 |
|----------------|---------|
| `BUS_FREE_WIFI`, `버스와이파이`, `city_bus` 류 | 시내버스 |
| `SKT_WiFi_PUBLIC`, `U+Zone`, `KT_GiGA_WiFi` | 지하철 역/차량 |
| `KTX_WiFi` | KTX |
| `SRT_WiFi` | SRT |

→ Wi-Fi SSID만으로도 대중교통 종류를 직접 확인할 수 있는 "한국만의 장점"

### 7-3. 교통카드 연동 가능성

- T-money, 캐시비 등 교통카드 이력 API가 있다면, 센서 분류 결과를 "검증/보정"하는 데 사용 가능
- NFC 태그 감지 시점 = 대중교통 탑승 시점으로 활용 가능

## 8. WB3 앱 구현 로드맵 제안

### Phase 1: 즉시 구현 가능 (1~2주)

**방법:** 플랫폼 내장 API만 사용
```
Google Activity Recognition API (Android)
Apple Core Motion (iOS)
```

**감지 가능:**
- ✅ 걷기 / 뛰기 / 자전거 / 정지
- ✅ "차량 탑승 중" (종류 미구분)
- ❌ 버스/승용차/지하철 개별 구분 불가

**정확도:** 85~90%
**배터리 영향:** 최소
**코드 난이도:** 쉬움

### Phase 2: GPS 속도 프로파일 추가 (2~4주)

**방법:** 가속도계 + GPS 조합
- 차량 탑승 감지 시에만 GPS 활성화 (듀티 사이클링)
- GPS 속도로 교통수단 후보군 좁히기

**추가 감지 가능:**
- ✅ KTX/SRT (속도 250+ km/h)
- ✅ 오토바이 (엔진 진동 패턴)
- △ 버스 vs 승용차 (정차 빈도로 어느 정도 구분)

**정확도:** 88~92%
**배터리 영향:** 중간 (GPS 간헐 사용)

### Phase 3: GIS 데이터 융합 (1~2개월)

**방법:** 한국 교통 GIS 데이터 연동
- 공공데이터포털 버스 정류장/노선 데이터
- 지하철역 위치 데이터
- Wi-Fi SSID 감지 추가

**추가 감지 가능:**
- ✅ 버스 (노선 매칭 + Wi-Fi)
- ✅ 지하철 (역 위치 + Wi-Fi + 기압)
- ✅ 기차 (역 위치 + 속도)

**정확도:** 90~93%
**배터리 영향:** 중간

### Phase 4: 커스텀 ML 모델 (3~6개월, 선택)

**방법:** TensorFlow Lite / Core ML 온디바이스 모델
- 한국 교통 환경 데이터로 학습
- CNN-BiLSTM 또는 TCN 아키텍처

**기대 정확도:** 92~95%
**배터리 영향:** 중간 (온디바이스 추론은 가벼움)

## 9. 실현 가능성 최종 평가

### 교통수단별 감지 난이도와 권장 전략

| 교통수단 | 감지 난이도 | 권장 방법 | 기대 정확도 |
|---------|-----------|---------|-----------|
| 걷기 | ⭐ 매우 쉬움 | 플랫폼 API | 90~97% |
| 뛰기 | ⭐ 매우 쉬움 | 플랫폼 API | 93~99% |
| 자전거 | ⭐⭐ 쉬움 | 플랫폼 API | 80~95% |
| 오토바이 | ⭐⭐ 보통 | 가속도계 진동 분석 | 75~85% |
| 승용차 | ⭐⭐⭐ 보통 | GPS 속도 + 소거법 | 80~88% |
| 버스 | ⭐⭐⭐⭐ 어려움 | GPS + GIS 노선 매칭 | 75~85% |
| 지하철 | ⭐⭐⭐ 보통 | 기압계 + GIS + Wi-Fi | 80~90% |
| KTX/SRT | ⭐⭐ 쉬움 | GPS 속도 (250+ km/h) | 90~95% |
| 택시 | ⭐⭐⭐⭐⭐ 불가 | 센서로 구분 불가 → 승용차로 통합 또는 앱 연동 | N/A |

### "집↔회사 출퇴근" 시나리오에서의 현실적 정확도

사용자가 집과 회사/학교를 설정해두면 출퇴근 경로가 반복되므로, **개인화 학습**이 가능합니다.

```
예시) 매일 아침 8시, 집 → 걸어서 → 지하철역 → 지하철 → 걸어서 → 회사
      → 며칠 반복되면 "이 시간, 이 경로 = 지하철 출퇴근" 학습 가능
```

**개인화 학습 적용 시:**
- 반복 경로의 교통수단 감지 정확도: **90~95%로 향상**
- 새로운 경로(처음 가는 곳): 일반 정확도 유지 (85~90%)
