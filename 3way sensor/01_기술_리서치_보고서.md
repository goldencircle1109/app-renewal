# 활동 인식(Activity Recognition) 기술 리서치 보고서

> 작성일: 2026-03-19
> 목적: 스마트폰 센서를 이용한 회원 활동(걷기/뛰기/자전거) 자동 인식 가능성 조사

---

## 1. 결론: 가능한가?

**YES - 충분히 가능합니다.**

| 항목 | 내용 |
|------|------|
| 걷기 인식 정확도 | 90~97% |
| 뛰기 인식 정확도 | 93~99% |
| 자전거 인식 정확도 | 80~95% |
| 필요한 추가 하드웨어 | 없음 (스마트폰 내장 센서만 사용) |
| ML 전문가 필요 여부 | 내장 API 사용 시 불필요 |

---

## 2. 어떤 센서를 사용하나?

스마트폰에는 이미 필요한 센서가 모두 내장되어 있습니다.

### 2-1. 가속도계 (Accelerometer) - 핵심 센서

3축(X, Y, Z) 방향의 가속도를 측정합니다. **활동 인식의 90% 이상을 담당**하는 가장 중요한 센서입니다.

```
걷기: 진폭 약 1~2g, 주파수 1.5~2.5Hz (보행 리듬)
뛰기: 진폭 약 2~4g, 주파수 2.5~3.5Hz (착지 충격이 강함)
자전거: 진폭 낮고 불규칙, 페달링 주기 0.5~1.5Hz
```

쉽게 설명하면:
- **걷기**: 폰이 규칙적으로 살짝살짝 흔들림
- **뛰기**: 폰이 규칙적으로 크게 흔들림 (발이 땅에 닿을 때 충격)
- **자전거**: 폰이 거의 안 흔들리지만 노면 진동이 전달됨

### 2-2. 자이로스코프 (Gyroscope)

회전 속도를 측정합니다. 가속도계만으로 헷갈리는 활동을 구분하는 **보조 역할**입니다.

### 2-3. GPS

속도 정보를 제공합니다. 자전거(15~30km/h) vs 뛰기(8~15km/h) vs 걷기(3~7km/h) 구분에 매우 효과적입니다.

**단점**: 배터리 소모가 크고, 실내에서는 사용 불가합니다.

### 2-4. 만보계 (Step Counter)

별도 저전력 칩으로 동작하여 배터리 소모가 거의 없습니다. 걸음수 + 걸음 빈도로 걷기/뛰기 구분을 보조합니다.

### 2-5. 기압계 (Barometer)

고도 변화를 감지합니다. 계단/언덕 구간 감지에 유용하지만, 모든 기기에 탑재되어 있지는 않습니다.

## 3. 플랫폼별 제공 API

### 3-1. Android - Activity Recognition API

Google Play Services에 포함된 **무료 API**입니다. Google 내부 ML 모델을 사용하므로 개발자가 직접 ML을 구현할 필요가 없습니다.

**인식 가능 활동:**
- `WALKING` (걷기)
- `RUNNING` (뛰기)
- `ON_BICYCLE` (자전거)
- `STILL` (정지)
- `IN_VEHICLE` (차량)
- `TILTING` (기울임)

**특징:**
- 각 활동마다 0~100 신뢰도(confidence) 값 반환
- 75% 이상이면 신뢰할 수 있는 결과
- Transition API 사용 시 배터리 소모 최소화
- Android 2.3 이상 지원

**필요 설정:**
```gradle
// build.gradle
implementation 'com.google.android.gms:play-services-location:21.0.0'
```
```xml
<!-- AndroidManifest.xml -->
<uses-permission android:name="android.permission.ACTIVITY_RECOGNITION" />
```

### 3-2. iOS - Core Motion (CMMotionActivityManager)

iOS 7부터 제공되며, **모션 코프로세서(M-시리즈 칩)** 에서 처리되어 배터리 소모가 매우 적습니다.

**인식 가능 활동:**
- `walking` (걷기)
- `running` (뛰기)
- `cycling` (자전거) - iOS 8부터
- `stationary` (정지)
- `automotive` (차량)

**특징:**
- 신뢰도 3단계: `.low`, `.medium`, `.high`
- 과거 7일치 데이터 조회 가능
- 앱이 백그라운드에 있어도 데이터 수집 가능
- CPU 사용 없이 전용 칩에서 처리 → 배터리 효율 최고

**필요 설정:**
```
Info.plist에 NSMotionUsageDescription 추가
```

### 3-3. 크로스플랫폼 라이브러리

| 프레임워크 | 라이브러리 | 설명 |
|-----------|-----------|------|
| React Native | `react-native-activity-recognition` | Android/iOS 모두 지원하는 통합 래퍼 |

## 4. 활동 분류가 작동하는 원리

### 4-1. 방법 1: 플랫폼 내장 API (가장 쉬움)

```
앱에서 API 호출 → OS가 센서 데이터 수집 → OS 내부 ML 모델이 분류 → 결과 반환
```

개발자가 할 일: API 호출 코드만 작성하면 됩니다.

### 4-2. 방법 2: 규칙 기반 (커스텀)

```
가속도 벡터 크기 = sqrt(x² + y² + z²) - 1g (중력 제거)

판별 규칙:
1. 평균 크기 < 0.1g → 정지
2. 1.5~3.5Hz 주파수 피크 + 크기 > 1.5g → 뛰기
3. 1.5~3.5Hz 주파수 피크 + 크기 < 1.5g → 걷기
4. GPS 속도 > 8km/h + 가속도 진폭 낮음 → 자전거
```

### 4-3. 방법 3: 온디바이스 ML (가장 정확)

공개 데이터셋(UCI HAR 등)으로 모델을 학습시켜 앱에 탑재합니다.
- Android: TensorFlow Lite (모델 크기 ~27KB까지 경량화 가능)
- iOS: Core ML

## 5. 정확도와 한계

### 5-1. 정확도 비교

| 방법 | 정확도 | 구현 난이도 | 구현 기간 |
|------|--------|-----------|----------|
| 플랫폼 내장 API | 80~90% | 쉬움 | 1~3일 |
| 규칙 기반 | 70~85% | 보통 | 1~2주 |
| 온디바이스 ML | 92~98% | 어려움 | 1~3개월 |
| GPS + 센서 융합 | 90~95% | 보통 | 2~3주 |

### 5-2. 알려진 한계점

| 한계 | 설명 | 해결책 |
|------|------|--------|
| 자전거 인식 어려움 | 폰 위치(주머니/핸들)에 따라 신호가 달라짐 | GPS 속도 보조 사용 |
| 느린 뛰기 vs 빠른 걷기 | 6~8km/h 구간에서 경계가 모호함 | 보폭/걸음 빈도 분석 추가 |
| 버스 vs 자전거 혼동 | 버스 진동이 자전거와 유사 | GPS 정류장 패턴 분석 |
| 실내 GPS 불가 | 건물 안에서는 GPS 신호 없음 | 가속도계만으로 폴백 |
| 폰 위치 문제 | 손/주머니/가방에 따라 신호 차이 | 위치 독립적 알고리즘 사용 |

### 5-3. 배터리 소모

| 방식 | 배터리 영향 | 비고 |
|------|-----------|------|
| iOS CMMotionActivityManager | 매우 낮음 | 전용 칩 처리 |
| Android Transition API | 낮음 | Google 권장 방식 |
| 가속도계 직접 폴링 (50Hz) | 중간~높음 | 커스텀 구현 시 |
| GPS 상시 활성화 | 높음 (시간당 5~10%) | 필요시만 사용 권장 |
